How do I prioritise a feature backlog?
ASSISTANT
Use the RICE framework for objective scoring: RICE = (Reach × Impact × Confidence) ÷ Effort • Reach — How many users affected per quarter? • Impact — Scale 0.25 (minimal) to 3 (massive). Be honest. • Confidence — % certainty in your estimates (80% = solid data, 50% = gut feel) • Effort — Person-weeks of engineering time For fintech specifically, add a Compliance column — anything touching payments or KYC gets a mandatory review regardless of RICE score. Tip: Run the scoring in a shared sheet so engineers and stakeholders can see the logic. Reduces 'HiPPO' decisions.

What's the best way to write a PRD?
ASSISTANT
Lean PRD structure (avoids 20-page documents nobody reads): 1. Problem statement (2 sentences) — What user pain are we solving? What evidence do we have? 2. Success metrics — 2–3 measurable outcomes (activation rate, D7 retention, support ticket volume) 3. User stories — "As a [persona], I want [action] so that [outcome]" 4. Scope: In / Out — Explicit list of what is NOT in v1 5. Open questions — Things Engineering/Design need to resolve before building 6. Timeline — Target ship date + key milestones Ask Claude: "Turn these bullet points into a concise PRD. Keep it under one page." Then refine.
YOU
How do I measure if a feature was successful?
ASSISTANT
Three layers of measurement: ▸ Adoption — Did users find and use it? (Feature reach %, time-to-first-use) ▸ Engagement — Did they come back to it? (Weekly active users of feature, depth of usage) ▸ Outcome — Did it move the needle on the metric you set out to improve? (Conversion rate, churn, NPS) For fintech: also track error rates, failed transactions, and support tickets related to the feature. Common mistake: Measuring adoption only and calling it a success. A feature used once that doesn't retain users hasn't solved the problem. Set your success criteria in the PRD before you build — not after you ship.
